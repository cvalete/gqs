Boas Práticas

Criação de Pendência​

Tipo de Pendência​

Define o fluxo de trabalho da pendência.

Se a demanda for:

  1. um novo sistema;
  2. um módulo novo;
  3. uma funcionalidade que pode ter N atividades, deve ser criada como Tipo ÉPICO.
caution

Se a demanda for de um PROJETO EXISTENTE, deve ser cadastrada dentro do projeto como ÉPICO;

caution

Se a demanda for de um PROJETO NOVO, deve ser cadastrada dentro do projeto como DEMANDAS como um PRÉ-PROJETO;

Resumo​

É o TÍTULO DA DEMANDA - deve ser breve, claro e definir o que foi solicitado pelo demandador. No caso de solicitação vir por Processo, colocar o número do processo no Resumo.

Epic Name​

Colocar um texto que facilite no momento de associar as tarefas necessárias para atender a demanda.

Componente​

Podem ser associados títulos classificadores da demanda tais como: número da Ordem de serviço associada.

Descrição​

Outras informações sobre o problemas, demanda ou requisito associado devem ser especificados neste campo. Anexos: devem ser anexadas TODAS as evidências que o usuário demandador enviou ou que o atendente gerou na atividade de atendimento. Por exemplo: telas, relatórios, atas de reunião, ofícios, autorizações, emails…

Data da Solicitação​

Data de entrada da atividade/demanda.

Tempo Estimado​

Este tempo deve ser colocado no formato W D H, e deve ser literalmente o tempo total estimado para realização da tarefa explícita neste Jira. Se não houver certeza desta estimativa no momento de cadastramento deve-se deixar este preenchimento para depois.

Inicio e Fim Previsto​

Nestes campos deve-se informar exatamente a data prevista, estes serão base para relatórios de atendimento a SLA e atraso no atendimento. Se não houver certeza desta estimativa no momento de cadastramento deve-se deixar este preenchimento para depois.

Órgão Principal​

É o órgão atendido na execução do Sistema Projeto, nunca esqueça de colocar o órgão correto. Muitos Jiras têm seu esforço atribuído "ERRONEAMENTE" à IplanRio.

Responsável​

Este campo é automaticamente preenchido com a credencial de quem está cadastrando a demanda, se não for este o casa deve-se atribuir o responsável correto.

Envolvidos​

Devem ser apontados os membros da equipe que devem acompanhar o ciclo da demanda.

Deve constar no Relatório de Realizações?​

O SIM deve ser marcada para aquelas demandas que trarão um VALOR IMPORTANTE A MUNICIPALIDADE OU AO CIDADÃO. Quando este campo for SIM, no campo VALOR GERADO deverá ser preenchido o valor gerado após atendimento à DEMANDA. As informações geradas a partir do preenchimento desses campos darão base para o Relatório Mensal de Realizações e às Publicações do Periódico Conexão Iplan.

Atividades Relacionadas​

Ao criar uma atividade relacionada DIRETAMENTE a uma demanda (ao épico), você DEVE colocar o ÉPICO no campo EPIC LINK da atividade.

info

LEMBRE-SE: Ao cadastrar as ATIVIDADES NECESSÁRIAS para atendimento da demanda, você deve criar um novo(s) jira(s):

  1. No campo EPIC LINK informar o Epic Name da Demanda;
  2. No campo ÓRGÃO PRINCIPAL informar o nome do órgão demandante;
  3. Esta demanda pode ser uma manutenção perfectiva, gerenciamento de projeto, uma história, etc
  4. Se cada novo jira relacionado a demanda necessitar de subtarefas, você deve criá-las e NÃO precisa relacionar ao ÉPICO.

Campo comentário​

O campo COMENTÁRIO existe para registro de SITUAÇÕES IMPORTANTE ou até mesmo para TROCA DE MENSAGENS de dúvidas ou PEDIDOS DE INFORMAÇÕES entre técnicos e/ou usuários.

No caso de trocas de informações entre técnicos da IplanRio, sugerimos que a mesma seja restrita aos técnicos.

Para isto, siga os seguintes passos:

  1. Ao criar ou editar um comentário aparecerá abaixo da caixa de texto, um botão com um cadeado.

  2. Ao clicar no cadeado, aparecerá a relação das equipes para as quais você pode restringir a visão do comentário.

  3. Selecione IPLANRIO. Pronto!!

  4. O restante do comentário você continuará fazendo como sempre fez.

info

Observações

  1. Essa mesma funcionalidade está presente também nos comentários do registro de trabalho.
  2. Cabe ao líder do projeto ou alguém indicado pela GTIS/DOP, definir os usuários integrantes de cada Função de Projeto listada ao clicar no cadeado.

Registro de Trabalho​

O registro das HORAS DE TRABALHO deve conter o tempo em cada atividade executada, e deve ser feito de preferência DIARIAMENTE, ao lado deve-se especificar qual atividade exatamente o colaborador realizou naquelas horas.

TIPOEXEMPLO
Se for uma reuniãoInformar o PROPÓSITO da reunião (sprint, levantamento de requisitos, gerencial…)
TEMPO GASTO: 2 horas
Reunião de Planejamento da Sprint 22
Se for uma manutençãoTEMPO GASTO: 4 horas
Correção na pesquisa de fornecedores (permitia realizar a pesquisa sem ter produto/ serviço selecionado) gerando erro no sistema – corrigido.
Se for uma ação de Gerenciamento de ProjetoTEMPO GASTO: 2 horas
1. Reunião com Gestor para informar o andamento do projeto;
2. Planejamento e acompanhamento da sprint com o time (apresentação das estórias após validação com o cliente);
3. Análise de impacto sobre mudanças no projeto;
4.Reunião com stakeholders.

Outras Dicas​

  1. Uma atividade/demanda que está com situação ABERTA ou PRIORIZADA NÃO pode ter data de início REAL, FINAL OU ESFORÇO.

  2. Uma atividade/demanda que está com situação SUSPENSA, mas tem algum registro de esforço, PRECISA ter DATA DE INÍCIO REAL.

  3. Uma atividade que está FECHADA NÃO pode ter esforço após a DATA DE FIM REAL.

  4. Uma atividade/demanda deve ser SUSPENSA quando por algum motivo o CLIENTE DESISTIU da solicitação. Recomendamos que esta atividade/demanda seja revista dentro de 6 meses para atualização, se necessário, da sua situação.

  5. Na CRIAÇÃO ou na CLONAGEM de uma atividade/demanda, os campos DATA DE INÍCIO PREVISTO, DATA DE FIM PREVISTO, DATA DE INÍCIO REAL e DATA DE FIM REAL serão retirados da tela. Ao gravar o registro, eles ficarão em branco. Somente o campo DATA DA SOLICITAÇÃO será preenchido automaticamente com a data de criação/clonagem. Para alterá-lo, a atividade/demanda deverá ser editada.

  6. Uma atividade/demanda deve ser CANCELADA quando, por algum motivo, o cliente desistiu da solicitação em CARÁTER DEFINITIVO.

  7. Uma atividade que está com situação Fechado, Cancelado ou Recusar não pode estar sem data de início real e data de fim real e não poderá mais ser reaberta!

  8. As datas de início e fim real também não poderão mais ser alteradas.

  9. Os esforços e comentários não poderão ser corrigidos, excluídos ou acrescentados.

  10. Ao efetivar o fechamento, a ferramenta validará se a data de solicitação, data de início e fim real estão preenchidas.

  11. Para atividades que estão sendo fechadas, a ferramenta verificará se há algum esforço cadastrado.

  12. Uma atividade/demanda precisa ter a data de início real anterior a data de fim real.

  13. Uma atividade/demanda precisa ter o esforço dentro do intervalo de data de início real e data de fim real.

  14. Atividades fechadas e canceladas precisam de data de fim real para serem faturadas!

  15. Uma atividade/demanda que está com situação cancelada mas tem algum registro de esforço precisa ter data de início real e final, pois constará no faturamento.

  16. Todas as atividades que compõem uma demanda precisam estar em situação final (fechado, cancelado) para que o esforço seja considerado no faturamento.
